# Roost Mac v1.3 — Product Description

## Purpose of this document

This document is a deep product and architecture snapshot of **Roost Mac at v1.3.0**. It is written for another AI that does **not** have access to the source code, but needs to understand what the app is, what it looks like, how it behaves, how the pieces fit together, and what the current product maturity feels like.

The goal is not just to list features. The goal is to communicate:

- what the app is trying to be
- what the user experiences screen by screen
- how the desktop app behaves on macOS
- how data flows through the system
- how the backend model is structured
- how the premium subscription layer works
- how the AI helper (“Hazel”) is currently used
- how all of the code paths work together as one product

This is effectively a high-resolution product brief plus architectural walkthrough for the shipping Mac app.

---

## 1. What Roost is

Roost is a **shared life dashboard for two people**. It is a desktop app, built for macOS first, and designed around the idea of a couple sharing a home and keeping that home running together.

At its core, Roost combines several shared-home functions into one calm, warm, attractive interface:

- shared shopping list
- shared expense tracking and settlement
- household budgeting
- chores and recurring chores
- shared calendar surfaces
- notifications and activity history
- a lightweight shared notes area (“Pinboard”)
- a menubar widget for quick access

The product is intentionally not positioned like a workplace productivity app. It is not trying to be a team operating system. It is closer to a **private household companion**: something a couple leaves installed, opens every day, and relies on for the low-friction maintenance of shared life.

The app’s tone is warm and domestic rather than corporate. The interface is designed to feel soft, native-adjacent, and emotionally calm. A lot of the value is not just in “what it can do,” but in **how gently and cohesively it does it**.

## 3. High-level product philosophy and feel

Roost’s design and implementation are built around a few strong principles that show up everywhere in the product:

### Built for a household, not for teams
The core object in the system is a **home**. Users belong to a home. Most data is shared at the home level. The product assumes trust, shared context, and collaborative domestic use.

### Calm and warm rather than loud
The UI uses soft surfaces, rounded cards, muted backgrounds, gentle transitions, and human language. Even premium gating and billing warnings are designed to feel calm rather than aggressive.

### Native-feeling Mac experience
Even though it is an Electron app, a lot of effort has gone into making it feel like a proper Mac app:

- hidden inset title bar
- dock icon and tray integration
- menubar panel
- macOS notifications
- deep-link handling via `roost://`
- DMG-based release pipeline

### Real-time shared state
The household is meant to feel live. If one partner changes something, the other partner should see it reflect quickly. Roost treats realtime sync as core product infrastructure, not a nice-to-have.

### Product warmth over sterile utility
The app includes subtle celebration, intelligent copy, animated empty states, “Hazel” AI moments, and aesthetically soft information cards. It is trying to feel like a well-considered home product, not a spreadsheet with buttons.

## 4. Overall technical shape

Roost Mac is split into three major layers:

### 1. Electron main process
This is the desktop shell. It owns:

- app startup
- windows
- tray and menubar behavior
- updater integration
- notifications
- protocol/deep-link handling
- secure access to secrets and APIs like Claude/Hazel and Stripe

Key file:
- `src/main/index.ts`

### 2. Electron preload bridge
This is the secure bridge between main and renderer. It exposes a tightly controlled `window.api` object so the React app can ask the desktop shell to do privileged things without getting direct Node access.

Key file:
- `src/preload/index.ts`

### 3. React renderer app
This is the full user interface. It handles routing, screens, components, hooks, state composition, and all user interaction.

Key area:
- `src/renderer/src/app/`

The backend is Supabase, with database tables, RLS, edge functions, and realtime channels. Data access is wrapped in React hooks, then adapted into UI-facing context for the renderer.

## 5. How the app launches and runs

When Roost starts, the desktop shell initializes first.

### Startup sequence
At startup, Electron does the following:

1. registers the custom protocol `roost://`
2. configures the auto-updater
3. creates the main application window
4. creates a separate menubar window
5. creates a tray item in the menu bar
6. registers IPC handlers for all privileged functions

This means the app is not just “a window.” It is a small macOS desktop system with:

- a normal app window
- a dock presence
- a tray presence
- a popover-like menubar panel
- deep-link routing support

### Main window
The main window is a polished desktop container for the full app.

It has:

- a hidden inset title bar for a cleaner Mac look
- minimum dimensions to preserve layout quality
- a custom icon
- no direct Node integration in the renderer
- context isolation and sandboxing for security

In development it loads a local Vite server. In production it loads the built renderer HTML.

### Menubar window
Roost also has a separate dedicated menubar UI. This is a frameless floating window positioned under the tray icon. It opens and closes like a compact companion panel.

This gives the app a second mode of use:

- full app window for deep work
- quick-access menubar panel for lightweight interaction

### Tray item
The tray item provides:

- quick visibility in the macOS menu bar
- click-to-toggle behavior for the menubar window
- a context menu with actions like opening the main app or quitting

This makes Roost feel more present in day-to-day Mac use, more like a living utility than a closed app you only launch occasionally.

## 6. Security model between desktop and UI

The renderer does **not** get direct access to Node or Electron internals. Instead, the preload bridge exposes a curated `window.api` object.

That bridge includes functions for:

- receiving auth deep-link callbacks
- triggering native notifications
- asking Hazel to normalize or categorize text
- requesting chore suggestions
- getting budget insights
- creating Stripe checkout and portal sessions
- exporting/opening calendar links
- opening the main app window
- listening for updater status
- checking/downloading/installing updates

This matters because any secret-bearing functionality stays in the main process. The renderer can request something, but it does not directly hold API secrets.

## 8. Root app composition

The app is assembled through nested providers and layouts.

At a product level, the renderer is built from a stack of responsibilities:

- authentication state
- home/member data
- shopping/expenses/chores/etc. hooks
- UI coordination contexts
- routed page rendering
- global shell elements like top bar, offline banner, notifications, and quick actions

One important architectural piece is `AppContext`, which acts as an adapter layer between raw backend hooks and the UI’s view of the world.

Key file:
- `src/renderer/src/app/context/AppContext.tsx`

This file creates one unified UI-facing model with concepts like:

- `currentUser`
- `partnerName`
- `householdName`
- shopping items
- expenses
- chores
- calendar events
- budget categories
- notifications
- activities
- next shop date
- loading states

This is crucial for understanding the product: the UI is not talking directly to the database in lots of scattered places. Instead, hooks gather data, `AppContext` reshapes it into the exact forms the interface wants, and pages consume that coherent product-layer context.

## 11. Authentication and first-run flow

Roost’s auth flow is built around the concept of a person joining or creating a **home**.

### Welcome page
The Welcome page is the app’s first impression. It functions as a branded entry point to authentication and account setup.

At a product level it introduces the app and routes users into the sign-in / join journey.

### Join page
The Join page is for entering a household through an invite code or deep-link flow.

This is a central piece of the product because Roost is not just single-user software. The shared-home model depends on a second person joining the same home.

### Setup page
The Setup page handles post-auth onboarding. This is where a newly authenticated user can complete the “becoming part of a household” process.

### Deep-link auth support
Electron listens for `roost://` URLs and forwards them into the renderer. This is used for OAuth callback handling and join links.

That makes the auth flow feel properly desktop-native rather than hacky browser handoff.

## 14. Expenses page

The Expenses page manages shared household spending.

### Core purpose
It tracks expenses between household members and keeps a live sense of who owes whom.

### What users can do
- add expenses
- categorize expenses
- mark recurring vs one-off expenses
- track who paid
- see shared balance outcomes
- delete expenses
- settle up

### Product model
This page is not trying to be full accounting software. It is a household-friendly shared expense tool. The model is much closer to “keep our shared spending clear” than “build a finance department ledger.”

### Free vs premium behavior
By v1.3, the Expenses page is also one of the key places where Roost Nest gating shows up.

For free households:
- only the most recent 30 days of expense history are fully available

For Nest households:
- full history is available

This means the Expenses page is both a functional feature and a premium conversion surface.

### Settlement flow
There is also settlement support, allowing one user to repay another and clear the balance in a structured way.

### UI feel
The expense experience is visual and warm, not dry. Balance cards, categories, and actions are all wrapped in the app’s softer design language.

## 36. Validation and safety model

The app uses Zod schemas heavily.

### Product significance of validation
Zod is not just a technical preference here. It helps ensure that product surfaces receive trustworthy, expected shapes of data.

That reduces the chance of silent corruption or weird UI states.

### Security model
The database uses row-level security so users only access data for the home they belong to.

This is central to Roost’s “private shared home” model.

The app assumes trust within a home, but strong isolation between homes.

The subscription flow can feel native and smooth in the app, while still being architecturally safe.
